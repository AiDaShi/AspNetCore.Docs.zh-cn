### <a name="framework-reference"></a>框架参考

通过上面列出的软件包之一提供的ASP.NET Core 的功能可作为共享框架的一`Microsoft.AspNetCore.App`部分提供。 共享框架** 是安装在计算机上并包括运行时组件和目标包的一组程序集（*.dll* 文件）。 有关详细信息，请参阅[共享框架](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)。

* 面向 `Microsoft.NET.Sdk.Web` SDK 的项目隐式引用 `Microsoft.AspNetCore.App` 框架。

  对于这些项目，不需要其他引用：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* 目标`Microsoft.NET.Sdk`或`Microsoft.NET.Sdk.Razor`SDK 的项目应向 添加`FrameworkReference`显式`Microsoft.AspNetCore.App`：

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>使用 Docker 与框架相关的生成

使用依赖于ASP.NET Core[共享框架](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)的包的控制台应用的与框架相关的生成可能会导致以下运行时错误：

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`是包含ASP.NET核心运行时的共享框架，仅存在于[dotnet/core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker 映像上。 3.0 SDK 通过不包括共享框架中可用的库的副本来减小使用 ASP.NET Core 的与框架相关的生成的大小。 这是高达 18 MB 的潜在节省，但它要求存在/安装ASP.NET核心运行时才能运行应用。

要确定应用在 ASP.NET Core 共享框架上是否有依赖项（直接或间接），请检查在应用生成/发布期间生成的*运行时 config.json*文件。 以下 JSON 文件显示对 ASP.NET 核心共享框架的依赖：

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

如果应用正在使用 Docker，请使用包含 ASP.NET酷睿 3.0 的基本映像。 例如，`docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0` 。

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>配置

将 Kestrel 配置迁移到 （ `ConfigureWebHostDefaults` *Program.cs*） 提供的 Web 主机生成器 ：

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

如果应用使用 手动创建主机，`HostBuilder`请调用`UseKestrel`中的`ConfigureWebHostDefaults`Web 主机生成器：

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="connection-middleware-replaces-connection-adapters"></a>连接中间件替换连接适配器

连接适配器 （`Microsoft.AspNetCore.Server.Kestrel.Core.Adapter.Internal.IConnectionAdapter`） 已从 Kestrel 中删除。 将连接适配器替换为连接中间件。 连接中间件类似于ASP.NET核心管道中的 HTTP 中间件，但对于较低级别的连接。 HTTPS 和连接日志记录：

* 已从连接适配器移动到连接中间件。
* 这些扩展方法的工作方式与早期版本的 ASP.NET Core 相同。 

有关详细信息，请参阅[Kestrel 文章的"侦听选项.协议"部分中的 TlsFilterConnectionHandler](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#listenoptionsprotocols)示例。

### <a name="transport-abstractions-moved-and-made-public"></a>传输抽象移动并公开

Kestrel 传输层已作为 `Connections.Abstractions` 中的公共接口公开。 作为这些更新的一部分：

* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions`和关联的类型已被删除。
* <xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions.NoDelay>已移动到<xref:Microsoft.AspNetCore.Server.Kestrel.Core.ListenOptions>传输选项。
* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions.Internal.SchedulingMode`从<xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions>中删除。

有关详细信息，请参阅以下 GitHub 资源：

* [客户端/服务器网络抽象（点网/阿斯普网核心#10308）](https://github.com/dotnet/AspNetCore/issues/10308)
* [实现新的基岩侦听器抽象，并在顶部重新镀凯斯特雷尔（点网/阿斯普内科#10321）](https://github.com/dotnet/AspNetCore/pull/10321)

### <a name="kestrel-request-trailer-headers"></a>凯斯特雷尔请求拖车头

对于面向早期版本的ASP.NET核心的应用：

* Kestrel 将 HTTP/1.1 块状的挂页标头添加到请求标头集合中。
* 在读取请求正文到末尾后，可以使用拖车。

这会引起一些对标题和拖车之间歧义的担忧，因此拖车已移动到 3.0 中的新集合`RequestTrailerExtensions`（ ）。

HTTP/2 请求预告片包括：

* ASP.NET酷睿 2.2 中不可用。
* 3.0 表示`RequestTrailerExtensions`，以 。

存在新的请求扩展方法来访问这些拖车。 与 HTTP/1.1 一样，在读取请求正文到末尾后，拖车可用。

对于 3.0 版本，可以使用`RequestTrailerExtensions`以下方法：

* `GetDeclaredTrailers`&ndash;获取列出正文`Trailer`后要预期的预告片的请求标头。
* `SupportsTrailers`&ndash;指示请求是否支持接收拖车标头。
* `CheckTrailersAvailable`&ndash;检查请求是否支持预告片，以及它们是否可供读取。 此检查不假定有要读取的预告片。 即使`true`此方法返回，也可能没有要读取的预告片。
* `GetTrailer`&ndash;从响应获取请求的尾随标头。 调用`SupportsTrailers``GetTrailer`之前检查，或者如果<xref:System.NotSupportedException>请求不支持尾随标头，则可能发生 。

有关详细信息，请参阅[将请求预告片放入单独的集合中（dotnet/AspNetCore #10410）。](https://github.com/dotnet/AspNetCore/pull/10410)

### <a name="allowsynchronousio-disabled"></a>允许禁用同步IO

`AllowSynchronousIO`启用或禁用同步 I/O API，如`HttpRequest.Body.Read`和`HttpResponse.Body.Write`。 `Stream.Flush` 这些 API 是导致应用崩溃的线程不足的来源。 在 3.0 中，默认情况下禁用 `AllowSynchronousIO`。 有关详细信息，请参阅[Kestrel 文章中的同步 I/O 部分](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#synchronous-io)。

如果需要同步 I/O，可以通过配置正在使用的服务器上`AllowSynchronousIO`的选项来启用它（例如，在调用`ConfigureKestrel`时，如果使用 Kestrel）。 请注意，服务器（Kestrel、HttpSys、TestServer 等）都有自己的`AllowSynchronousIO`选项，不会影响其他服务器。 使用`IHttpBodyControlFeature.AllowSynchronousIO`以下选项，可按请求为每个服务器启用同步 I/O：

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

如果在 Dispose 中<xref:System.IO.TextWriter>调用同步 API 的实现或其他流时遇到[Dispose](/dotnet/standard/garbage-collection/implementing-dispose)问题，请改为调用<xref:System.IO.Stream.DisposeAsync*>新的 API。

有关详细信息，请参阅[[通知] 允许在所有服务器（点网/阿斯普内科#7644）中禁用同步IO。](https://github.com/dotnet/AspNetCore/issues/7644)

### <a name="microsoftaspnetcoreserverkestrelhttps-assembly-removed"></a>微软.AspNetCore.Server.Kestrel.https 程序集已被删除

在ASP.NET核心2.1中，*微软.AspNetCore.Server.Kestrel.https.dll*的内容被移到*微软.AspNetCore.Server.Kestrel.Core.dll。* 这是使用`TypeForwardedTo`属性的非中断更新。 对于 3.0，空的*Microsoft.AspNetCore.Server.Kestrel.https.dll*程序集和 NuGet 包已被删除。

引用[Microsoft.AspNetCore.Server.Kestrel.https](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)的库应更新ASP.NET核心依赖项更新为 2.1 或更高版本。

面向ASP.NET Core 2.1 或更高版本的应用和库应删除对[Microsoft.AspNetCore.Server.Kestrel.https](https://www.nuget.org/packages/Microsoft.AspNetCore.Server.Kestrel.Https)包的任何直接引用。

## <a name="routing-startup-code"></a>路由启动代码

* 如果应用实现`IRouter`或继承 ，`Route`请使用[DynamicRouteValue变形金刚](https://github.com/dotnet/AspNetCore.Docs/issues/12997)作为替换。

终结点路由支持与 的相同路由模式语法和路由模式创作功能`IRouter`。 终结点路由支持`IRouteConstraint`。 终结点路由支持`[Route]` `[HttpGet]`，和其他 MVC 路由属性。

### <a name="security-middleware-guidance"></a>安全中间件指南

围绕[中间件](xref:fundamentals/middleware/index)方法统一支持授权和 CORS。 这允许在这些方案中使用相同的中间件和功能。 此版本中提供了更新的授权中间件，并增强了 CORS 中间件，以便了解 MVC 控制器使用的属性。

#### <a name="cors"></a>CORS

以前，CORS 可能难以配置。 提供中间件用于某些用例，但 MVC 筛选器旨在在其他用例**中不使用**中间件。 使用 ASP.NET 酷 3.0，我们建议所有需要 CORS 的应用与端点路由一起使用 CORS 中间件。 `UseCors`可以提供默认策略，并且`[EnableCors]`属性`[DisableCors]`可用于在需要时覆盖默认策略。

* 类`MyController`禁用具有属性的`[DisableCors]`CORS。

```csharp

{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

#### <a name="async-suffix-removal-from-controller-action-names"></a>从控制器操作名称中删除异步后缀

在ASP.NET核心 3.0 中，ASP.NET核心`Async`MVC 从控制器操作名称中删除后缀。 路由和链路生成都受此新默认值的影响。 例如：

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

在 ASP.NET 核心 3.0 之前：

* 可以在*产品/ListAsync*路由上访问上述操作。
* 需要指定后缀的链接`Async`生成。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

在ASP.NET核心 3.0 中：

* 可以在 *"产品/列表*"路由上访问上述操作。
* 链接生成不需要指定`Async`后缀。 例如：

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

此更改不会影响使用 属性[`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute)指定的名称。 可以使用 中的以下代码禁用默认行为`Startup.ConfigureServices`：

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>链接生成更改

如在有关[与早期路由版本差异](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)的文档中所述，在链接生成（例如使用`Url.Link`和类似的 API）上存在一些差异。 其中包括:

* 默认情况下，当使用端点路由时，生成的 URI 中路由参数的大小写不一定保留。 此行为可以通过`IOutboundParameterTransformer`接口进行控制。
* 为无效路由（不存在的控制器/操作或页面）生成 URI 将在终结点路由下生成空字符串，而不是生成无效 URI。
* 环境值（当前上下文中的路由参数）不会自动用于与端点路由的链接生成。 以前，在生成指向其他操作（或页面）的链接时，将从*当前*路由环境值推断出未指定的路由值。 使用端点路由时，必须在链接生成期间显式指定所有路由参数。

### <a name="razor-pages"></a>Razor 页面

映射剃刀页面现在发生在内部`UseEndpoints`。

如果`MapRazorPages`应用使用剃刀页面，则添加。 由于端点路由包括对许多框架的支持，因此现在选择添加 Razor 页面。

在以下`Startup.Configure`方法中，`MapRazorPages`添加对 Razor 页面的支持：

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapRazorPages();
    });
}
```

运行状况检查可用作端点路由*的路由器软件*。

添加`MapHealthChecks`以将运行状况检查与终结点路由一起添加。 该方法`MapHealthChecks`接受类似于`UseHealthChecks`的参数。 使用`MapHealthChecks`超过`UseHealthChecks`的优点是能够应用授权和对匹配策略具有更大的细粒度控制。

## <a name="opt-in-to-runtime-compilation"></a>选择加入运行时编译

在ASP.NET Core 3.0 之前，视图的运行时编译是框架的一个隐式功能。 运行时编译补充视图的生成时间编译。 它允许框架在修改文件时编译 Razor 视图和页面 *（.cshtml*文件），而无需重建整个应用程序。 此功能支持在 IDE 中快速编辑和刷新浏览器以查看更改的方案。

在ASP.NET酷3.0中，运行时编译是一种选择加入方案。 生成时间编译是默认情况下启用的视图编译的唯一机制。 运行时依赖于 Visual Studio 代码中的 Visual Studio 或[点网监视](xref:tutorials/dotnet-watch)，以在检测到*对 .cshtml*文件的更改时重建项目。 在 Visual Studio 中，对 *.cs、.cshtml*或 *.razor*文件所做的更改将发生在正在运行的项目中<kbd>（Ctrl_F5），</kbd>但不调试<kbd>（F5），</kbd>从而触发项目的重新编译。 *.cs*

要在 ASP.NET Core 3.0 项目中启用运行时编译，请：

1. 安装[微软.AspNetCore.Mvc.Razor.运行时编译](https://www.nuget.org/packages/Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation)NuGet包。
1. 更新`Startup.ConfigureServices`以调用`AddRazorRuntimeCompilation`：

    对于ASP.NET核心 MVC，请使用以下代码：

    ```csharp
    services.AddControllersWithViews()
        .AddRazorRuntimeCompilation(...);
    ```
 
    对于ASP.NET核心剃刀页面，请使用以下代码：
 
    ```csharp
    services.AddRazorPages()
        .AddRazorRuntimeCompilation(...);
    ```
 
中的https://github.com/aspnet/samples/tree/master/samples/aspnetcore/mvc/runtimecompilation示例显示了在开发环境中有条件地启用运行时编译的示例。

有关 Razor 文件编译的详细信息，请参阅<xref:mvc/views/view-compilation>。

## <a name="migrate-libraries-via-multi-targeting"></a>通过多目标迁移库

库通常需要支持 ASP.NET 核心的多个版本。 大多数根据以前版本的ASP.NET Core 编译的库应继续工作，没有问题。 以下条件要求交叉编译应用：

* 库依赖于具有二进制[中断更改](#breaking-api-changes)的功能。
* 该库希望利用 ASP.NET Core 3.0 中的新功能。 

例如：

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFrameworks>netcoreapp3.0;netstandard2.0</TargetFrameworks>
  </PropertyGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netcoreapp3.0'">
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="Microsoft.AspNetCore" Version="2.1.0" />
  </ItemGroup>
</Project>
```

用于`#ifdefs`启用ASP.NET核心 3.0 特定的 API：

```csharp
var webRootFileProvider =
#if NETCOREAPP3_0
    GetRequiredService<IWebHostEnvironment>().WebRootFileProvider;
#elif NETSTANDARD2_0
    GetRequiredService<IHostingEnvironment>().WebRootFileProvider;
#else
#error unknown target framework
#endif
```

有关在类库中使用ASP.NET核心 API 的详细信息，请参阅<xref:fundamentals/target-aspnetcore>。
